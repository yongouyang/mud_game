## 4. 架构对比

| 维度 | oiuv_mud | mud_game |
|------|----------|----------|
| **代码组织** | 每个对象一个 .c 文件 (LPC) | 数据驱动 JSON + TypeScript 系统类 |
| **数据定义** | 分布在每个 .c 文件内部 | 集中在 `server/src/data/*.json` |
| **系统逻辑** | 守护进程 (daemon) 模式 | System 类 + CommandRouter |
| **扩展新内容** | 需编写 LPC 代码 | 编辑 JSON + 可能少量 TS |
| **可测试性** | eval 命令 (admin only) | **436 个自动化测试** ✅ |
| **可部署性** | 手动编译 + 启动 | **Docker + Terraform (CI/CD ready)** ✅ |
| **客户端** | 专用 Mudlet/telnet | **浏览器零安装** ✅ |
| **HTTPS** | ❌ | ✅ (ALB 终结) |
| **水平扩展** | ❌ 单进程 | ✅ Fargate 可扩 |
| **断线重连** | ❌ | ✅ Socket.io 自动 |

> ✅ **mud_game 在工程化方面远胜原版**：自动化测试、容器化部署、浏览器即客户端、CI/CD 管道就绪。

## 8. 推荐下一步路线

> 以下建议顺次执行，每轮约 3-7 天。

### 🥇 Phase A：内容质量打磨（第一优先级，最小代价最大收益）

当前最大矛盾：**数据量够了，但质量不足**。734 武功和 1,638 NPC 中大量是"占位符"。

| # | 任务 | 工作量 |
|---|------|--------|
| A1 | **武功去重与行为差异化** — 清理重复/占位符武功；为 39 个门派各设计 3-5 个独特 perform | 中 |
| A2 | **NPC 行为优化** — 给 NPC 分配正确的 roomId，确保每个门派大厅 ≥1 掌门+2 弟子 | 中 |

## 9. 下一步具体行动建议

### 推荐路线：**Phase A（内容质量打磨）→ Phase B（社交）→ Phase C（长期目标）**

理由：
1. **Phase A** 代价最小、见效最快——让现有 734 武功 / 1,638 NPC 从"有量无质"变成"真正可玩"
2. **Phase B** 改变游戏的基本体验模型——从单机到多人在线
3. **Phase C** 给玩家一个"为什么继续玩"的理由——转世、竞技、传奇装备的长线追求

### 如果只想做一件事，先做这个：

> **聊天系统 (Phase B1)** — 仅仅实现 `say`/`tell`/`shout`/门派频道，就能让多人同时在线的体验质变。当前 39 门派、战斗系统已经足够单机娱乐，缺的就是彼此看见彼此说话的基本社交。
