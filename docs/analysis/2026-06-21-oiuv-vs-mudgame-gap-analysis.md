| Attribute | Key |
|---|---|
| 臂力 | str |

**Missing**: `per` (容貌), `kar` (福缘), `neili` (内力), `jingli` (精力)

### Critical mechanism gap: Skill-based attribute improvement
In oiuv_mud, attributes are **not static** — they improve based on highest related skill level:
```
str = base_str + (highest_of_unarmed_cuff_finger_strike_hand_claw / 10) + tattoo_bonus + reborn_bonus
int = base_int + (literate / 10) + tattoo_bonus + reborn_bonus
con = base_con + (force / 10) + tattoo_bonus + reborn_bonus
dex = base_dex + (dodge / 10) + tattoo_bonus + reborn_bonus
per = base_per + tattoo_bonus - age_penalty (decays after 30)
```
In mud_game, `player.attributes` are static — they never change from skill advancement.

## 3. Combat System Gap

| Mechanism | oiuv_mud | mud_game | Status |
|---|---|---|---|
| 4-layer defense chain (parry→dodge→force→HP) | ✅ | ✅ | ✅ MATCH |
| Multi-enemy combat | ✅ Up to 4 opponents | ❌ 1v1 only | 🔴 GAP |
| Guarder system | ✅ NPCs protect each other | ❌ | 🔴 GAP |
| Killer tracking | ✅ killer list + want_kills list | ❌ | 🔴 GAP |
| Kill vs knockout | ✅ DPS (defeated) vs PKS (killed) distinction | ❌ HP=0 = dead | 🔴 GAP |
| Shen (善恶/align) system | ✅ Derived from killer tracking | ❌ | 🔴 GAP |
| Auto-combat heartbeat | ✅ 1s heart_beat | ✅ Scheduler-driven auto ticks | ✅ MATCH |
| Poison-on-hit | ✅ poisonChance + poisonLevel | ✅ poisonChance + poisonLevel | ✅ MATCH |
| Critical hits | ✅ 10% 1.8x | ✅ 10% 1.8x | ✅ MATCH |
| Speed formula | ✅ dex-based | ✅ dex + dodgeLevel-based | ✅ MATCH |

### Combat damage formula comparison
```
oiuv_mud:  baseDmg = skill_damage + str_bonus + weapon_damage
           finalDmg = baseDmg × random(0.8~1.2) × crit(1.8 if 10%)
           Apply parry → dodge → force absorb → HP

mud_game:  strikeDmg = def.damageBase + def.damageScale × level
           Apply parry → dodge → force absorb → HP
           crit: 10% 1.8x (partially implemented)
```

## 12. Architecture Preservation Notes

When porting systems from oiuv_mud to mud_game, maintain these invariants:

1. **Data-driven design**: All game content stays in JSON files under `server/src/data/`
2. **System separation**: Each game system is a standalone TypeScript class under `server/src/systems/`
3. **Command routing**: All player commands route through `CommandRouter.ts`
4. **Event-driven WebSocket**: Server pushes events to clients via `socket.emit('output', ...)`
5. **Testable**: Every system must have unit tests (`*.test.ts`)
6. **Formula fidelity**: Combat formulas, skill damage calculations, and level curves should match oiuv_mud exactly
7. **Scheduler-based time**: All periodic effects use the Scheduler, not `setInterval`
